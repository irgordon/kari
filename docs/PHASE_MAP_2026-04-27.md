# Kari Phase Map - 2026_04_27 — Authoritative Multi‑Phase Roadmap
This document defines the only valid phase sequence for the Kari Orchestration Engine.
All agents, contributors, and automated systems must treat this file as canonical.

## Authority Boundary
If a defect prevents completion of a phase, the phase may pause but not be skipped.
Remediation must occur within the current phase unless a formal revision updates this document.
No work from a later phase may begin until the current phase’s exit criteria are fully satisfied.

---

# Phase 3–N Roadmap (Post‑Audit Baseline)

This roadmap assumes:
- React migration complete
- Concurrency & Correctness Audit complete
- No correctness fixes applied yet
- Go module + proto surfaces not yet normalized

The ordering below is mandatory. Correctness work must not begin until Phase 3 is complete.

---

# Phase 3 — Build & Contract Baseline (A1/A3)
Purpose: Establish a deterministic, reproducible, contract‑stable foundation.
No correctness fixes. No behavioral changes. No refactors.

## 3.1 Go Module Normalization (A1)
- Rewrite all internal imports to match `go.mod` module path.
- Remove stale import roots.
- Run `go mod tidy`.
- Ensure reproducible builds.

## 3.2 Proto Contract Normalization (A2/A3)
- Regenerate Go and Rust protobuf stubs.
- Ensure deterministic output.
- Commit generated artifacts.
- Ensure `make proto-check` passes.

## 3.3 Build Baseline
From a clean checkout:
- `go build ./...` passes.
- `go test ./...` passes, or failures are cataloged as pre‑existing.
- All known failing tests must be recorded in a tracked issue list.
- No new test failures may be introduced during Phase 3.
- Rust agent builds cleanly.
- React frontend builds cleanly.
- No proto drift.
- No module drift.

## Phase 3 Exit Criteria
- All builds pass from a clean checkout.
- `go test ./...` runs successfully or failures are documented with traceable issue IDs.
- `make proto-check` passes.
- Rust and frontend builds pass.
- No correctness fixes applied.
- No behavioral changes made.
- Phase 3 guarantees become baseline invariants for all subsequent phases.

# Phase 4 — Correctness & Contract Fixes (Audit‑Driven)
Purpose: Apply correctness fixes identified in the audit.
Now safe because contracts and modules are stable.

## Phase 4A — Required Fixes (must pass for Phase 4 exit)

### Backend (Go)
- DI‑1: Refresh token scheme unified.
- RC‑1: Rate‑limiter `visitor.lastSeen` race fixed.
- DI‑2: SSH key AAD bound to real application ID.
- DI‑3: Status update before broadcast.
- OV‑1: UUID trace IDs.
- CB‑1: Single Hub‑backed SSE path.

### Agent (Rust)
- CB‑5: `start_command` sanitization before systemd templating.

## Phase 4A Exit Criteria
- All required fixes implemented and validated.
- No regressions in build or proto surfaces.
- Phase 4 guarantees become baseline invariants.

---

## Phase 4B — Optional / Deferrable Fixes
These are correctness‑adjacent or defense‑in‑depth improvements.

- CB‑3: Local gRPC credentials.
- CB‑2: SSE disconnect cleanup improvements.
- FS‑1: TOCTOU mitigation beyond documentation.
- RC‑2: Load hardening for system monitor (only if testing shows starvation).

These may be scheduled in Phase 6 or Phase 7 unless testing proves otherwise.

# Phase 5 — Verification Gate (Test‑Driven Acceptance)
Purpose: Prove correctness fixes are real, stable, and regression‑safe.

## 5.1 Go Verification
- `go test -race ./...` passes cleanly.
- Integration tests:
  - Refresh token lifecycle.
  - Deployment success/failure.
  - Trace ID uniqueness.
  - SSE log consistency.

## 5.2 Rust Verification
- Unit tests for systemd sanitization.
- Stress tests for GetSystemStatus.
- Fuzz tests for start_command and jail paths.

## 5.3 Cross‑Boundary Verification
- End‑to‑end deployment → logs → status transitions.
- SSE cancellation and reconnect.
- gRPC stream cancellation behavior.

## Phase 5 Exit Criteria
- All verification tests pass from a clean checkout on CI without manual intervention.
- Race detector clean.
- All integration tests green.
- No correctness regressions.
- SSE/log consistency validated.
- Phase 5 guarantees become baseline invariants.

---

# Phase 6 — Integration Reliability Hardening
Purpose: Ensure the system behaves correctly under real‑world failure modes.

## 6.1 API ↔ Agent End‑to‑End Harness
- Deterministic test harness for deployments, logs, and status transitions.

## 6.2 Failure Mode Testing
- Agent crash mid‑deployment.
- Unix socket loss.
- systemd failure to start/stop.
- Partial log delivery.
- Network jitter and reconnect.

## 6.3 Streaming Robustness
- SSE reconnect behavior.
- Cancellation propagation.
- Log completeness under churn.

## Phase 6 Exit Criteria
- All failure modes tested and documented.
- No liveness or correctness gaps remain.
- Phase 6 guarantees become baseline invariants.

# Phase 7 — Operational Readiness
Purpose: Make Kari production‑operable with clear SLOs, metrics, and runbooks.

## 7.1 Telemetry & SLOs
- Deployment success rate.
- Health check latency.
- Log delivery completeness.

## 7.2 Observability
- Metrics for deployments, agent health, SSE streams, error rates.

## 7.3 Runbooks
- Deployment failures.
- Agent crashes.
- systemd failures.
- Log inconsistencies.
- Token refresh failures.

## 7.4 CI/CD Release Gates
- Race detector.
- Proto drift.
- Integration tests.
- Contract tests.

## Phase 7 Exit Criteria
- System is operable with measurable SLOs.
- CI/CD enforces correctness.
- Phase 7 guarantees become baseline invariants.

---

# Phase 8+ — Evolution
Only after Phases 3–7 are complete do we consider:
- New features.
- Architectural changes.
- Performance optimizations.
- Plugin systems.
- Multi‑agent orchestration.
- Multi‑cluster support.

All future work must preserve the correctness guarantees established in Phases 3–7.
