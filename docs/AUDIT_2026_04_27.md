# Kari Concurrency & Correctness Audit Report — Phase A1 - 2026-04-27

## 1. Introduction & Scope

This report covers all concurrency-relevant code paths in the Kari codebase across:

- Go API (`api/`) — goroutines, sync primitives, HTTP handlers, background workers  
- Rust Agent (`agent/`) — tokio async executor, Arc/Mutex, gRPC streaming, system-level operations  
- TypeScript / React Frontend (`frontend/`) — event loop, SSE, async state management  
- Cross-boundary interactions — Go ↔ Rust over Unix Domain Socket gRPC, Hub ↔ SSE, JWT / Opaque token lifecycle  

---

## 2. Concurrency Model Overview

| Layer              | Model                    | Primitives Used |

| Go API             | goroutines + channels    | sync.RWMutex, sync.Map, sync/atomic, context |

## 3.1 Race Conditions

### RC-1 [HIGH] — Unsynchronized read/write on `visitor.lastSeen`

File:

```

api/internal/api/middleware/auth.go

```

Problem:

Concurrent goroutines access a shared visitor struct without synchronization.

```

vis.lastSeen = time.Now()

```

Background cleanup goroutine reads the same field without locking.

Risk:

- Data race detectable by Go race detector  
- Potential torn read on multi-word `time.Time`  
- Incorrect rate-limiter cleanup behavior  

Fix:

Add:

```

sync.RWMutex

```

or:

```

atomic.Pointer[time.Time]

```

---

### RC-2 [LOW] — Blocking Mutex inside async handler

File:

```

agent/src/server.rs

```

Problem:

```

std::sync::Mutex

```

used inside tokio async handler.

Risk:

- Thread starvation under load  
- Reduced async scheduler responsiveness  

Fix:

Use:

```

tokio::sync::Mutex

```

or:

```

spawn_blocking()

```

---

## 3.3 Data Inconsistencies / Corruption

---

### DI-1 [CRITICAL] — Incompatible refresh token schemes

Files:

```

auth_service.go
token_service.go

```

System behavior:

- Token generator creates opaque tokens
- Token verifier expects JWT

Result:

Refresh endpoint always fails.

Impact:

Session renewal permanently broken.

Fix:

Choose one scheme:

Preferred:

Opaque token lookup using SHA-256 hash.

---

### DI-2 [MEDIUM] — Ephemeral AppID breaks SSH key binding

File:

```

deployment_handler.go

```

Problem:

New UUID generated per deployment request.

Result:

Previously stored encrypted SSH keys become unrecoverable.

Fix:

Use persisted application ID as encryption associated data.

---

### DI-3 [LOW] — Status update occurs after broadcast

File:

```

deployment_worker.go

```

Current order:

```

Broadcast
UpdateStatus

```

Correct order:

```

UpdateStatus
Broadcast

```

---

## 3.6 Order Violations

---

### OV-1 [MEDIUM] — Non-deterministic trace ID generation

File:

```

application_service.go

```

Problem:

Trace ID depends on optional context value.

Risk:

Log collisions under concurrent deployments.

Fix:

Use:

```

uuid.New().String()

```

---

### OV-2 [LOW] — Cancel registered before stream creation

File:

```

deployment_worker.go

```

Risk:

Premature deployment failure.

Fix:

Register cancel handler after stream establishment.

## 4. Cross-Boundary Correctness

---

### CB-1 [MEDIUM] — Dual SSE implementations

Files:

```

sse_handler.go
deployment_handler.go

```

Risk:

Duplicate gRPC streams per client.

Possible divergence between UI and database logs.

Fix:

Maintain a single canonical SSE path.

---

### CB-2 [LOW] — Non-preemptive disconnect detection

File:

```

sse_handler.go

```

Risk:

Delayed disconnect handling.

---

### CB-3 [LOW] — Insecure gRPC credentials

File:

```

kari-api/main.go

```

Risk:

Plaintext gRPC messages over Unix socket.

Fix:

Use:

```

credentials/local

```

Defense-in-depth improvement.

---

### CB-5 [MEDIUM] — Unsanitized systemd command input

File:

```

systemd.rs

```

Fix:

Sanitize command string before templating.
